对于condition_variable理解

condition_variable的wait做了什么
所以第25行lock上锁后,执行到wait时,本线程一定持有锁,如果wait检查到条件不满足,就会释放锁,然后等待唤醒,如果他被唤醒了,就会一直尝试获取锁,所以wait返回后一定是持有锁,并且条件满足的情况

  1. wait 前
    lock 已经上锁，当前线程持有 mutex m。
  2. wait 时
    cv.wait(lock, pred) 首先检查 pred（谓词）是否满足。
    如果 条件不满足，wait：
    自动释放 mutex（unlock），让其它线程可以 push/notify。
    进入等待状态（阻塞）。
  3. 唤醒后
    当有 thread 执行 cv.notify_one() 或 notify_all()：
    被唤醒的线程会重新尝试获得 mutex（lock m），直到锁可用为止。
    一旦获得锁，wait 返回前（此时还没有离开 wait），lock 已经重新持有 mutex。
  4. wait 返回
    谓词 pred 会再次被检查（唤醒时会自动 re-check）。
    只有谓词为真（条件满足），wait 才会真正返回到后续代码，否则继续再睡——这是所谓的 “条件变量虚假唤醒安全”，保证没满足条件不会进入后续逻辑。
    所以，wait 返回后，当前线程持有锁，且条件谓词已经为真。

mutex互斥量

lock_guard和unique_lock.

lock_guard

  • lock_guard 是 C++ 标准库中的互斥量封装类，用于保护共享数据。
  • 当 lock_guard 的构造函数被调用时，互斥量会被自动锁定。
  • 析构函数被调用时，互斥量会被自动解锁。
  • lock_guard 对象不能复制或者移动，只能在局部作用域使用。

unique_lock

  • unique_lock 是互斥量封装类，也用于对互斥量进行加锁/解锁。
  • 构造时自动加锁，析构时自动解锁。

方法说明

  • lock()：尝试对互斥量进行加锁。如果互斥量被其他线程持有，会一直阻塞。
  • try_lock()：尝试对互斥量进行加锁。如果互斥量被其他线程占用，立刻返回 false，否则加锁并返回 true。
  • try_lock_for(const std::chrono::duration<Rep,Period>& rel_time)：尝试对互斥量进行加锁，在超时时间内尝试加锁。超时后返回 false，否则加锁并返回 true。
  • try_lock_until(const std::chrono::time_point<Clock,Duration>& abs_time)：尝试对互斥量进行加锁，在超时时间内尝试加锁。超时后返回 false，否则加锁并返回 true。
  • unlock()：手动解锁。

注意事项

  • 如果对自己已经持有的锁调用 try_lock，会抛出异常 std::system_error，提示 Resource deadlock avoided。

  • C++ 的互斥量帮助解决多线程程序中的共享数据竞争问题，实现线程同步。

call_once实现单例模式

确保多线程下某个函数只被调用一次

  • std::call_once 是 C++ 标准库的一个函数，确保某个函数只被调用一次
  • 原型：
  • flag 是一个 once_flag 对象，标记函数是否已经被调用
  • func 是需要被调用的函数或者可以调用的对象，args 是函数或者可调用对象的参数
  • 在多个线程中同时调用 call_once，只有一个线程能成功执行 func 函数，其他线程会等待该函数执行完成
  • call_once 只能在线程里面使用，main 里面调用会报错

async future promise异步api

  • async 是 C++11 引入的一个函数模板，用于异步执行一个函数，返回一个 future 对象，表示异步操作的结果。
    它避免了手动创建和管理线程的麻烦。

  • packaged_task 是一个类模板，
    用于把一个可调用对象封装成一个异步操作，并返回一个 future，表示异步操作的结果。

  • promise 是一个类模板，
    用于在一个线程中产生一个值，在另一个线程中获取。
    promise 通常和 async 以及 future 一起使用，用于实现异步编程。
